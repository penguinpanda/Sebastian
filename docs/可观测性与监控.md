# Sebastian 可观测性与监控

更新时间: 2026-06-11

## 1. 当前能力

- 请求级 trace middleware（`x-trace-id` 透传/生成）。
- MCP invoke 路径注入 trace_id。
- Agent 工具调用日志写入 trace_id。
- Celery 链路透传 trace 并记录执行日志。
- readiness 接口支持依赖异常时返回 503。

## 2. Trace 规范

### 2.1 Header 约定

- 请求头: `x-trace-id`
- 若未传入，系统自动生成。

### 2.2 链路要求

必须覆盖以下节点:

1. API 入口
2. Agent 编排节点
3. MCP 工具调用
4. Celery 入队与执行
5. 任务历史查询接口

## 3. 日志规范

建议统一结构化字段:

- `timestamp`
- `level`
- `trace_id`
- `path`
- `method`
- `task_id`（可选）
- `user_id`（可选）
- `error_code`（失败时）

## 4. 监控最小指标（Dev Ready）

当前项目未集成完整 metrics 导出，先执行最小可观测检查:

1. API 健康状态可查询。
2. Agent 请求失败可通过日志定位。
3. Celery 执行历史可通过任务历史 API 检索。
4. readiness 能准确反映依赖不可用。

## 5. 建议下一步（P2）

1. 引入 Prometheus metrics（请求耗时、错误率、队列长度）。
2. 统一日志输出到 JSON 并接入集中检索。
3. 定义告警阈值:
   - readiness 连续失败
   - agent chat 错误率超阈值
   - celery 队列堆积

## 6. 排障入口

- 健康检查: `/api/health` `/api/health/dependencies` `/api/health/readiness`
- Agent 状态: `/api/agent/tasks/{task_id}`
- 队列状态: `/api/agent/queue/stats`
- 任务历史: `/api/tasks/history/{task_id}` 和 `/api/tasks/history`
