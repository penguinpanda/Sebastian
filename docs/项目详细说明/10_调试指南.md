# 调试指南

> 新人最需要：一个请求失败了怎么办。

## 调试总流程

一个请求失败时，按以下顺序排查：

```
1. API 收到请求了吗？
       ↓
2. Agent 执行了吗？
       ↓
3. Graph 节点有没有走？
       ↓
4. Service 有没有返回？
       ↓
5. DB / ES / Redis 有没有数据？
       ↓
6. LLM 调用成功了吗？
```

---

## 第 1 步：API 收到请求了吗？

### 健康检查

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/health
Invoke-RestMethod http://127.0.0.1:8000/api/health/readiness
Invoke-RestMethod http://127.0.0.1:8000/api/health/dependencies
```

- `health` 返回 `{"status":"ok"}` → API 进程正常
- `readiness` 返回 503 → 依赖（PG/Redis/ES）不可用
- `dependencies` 显示各依赖连接状态

### 检查 trace_id

每个请求响应头包含 `x-trace-id`：

```powershell
$response = Invoke-WebRequest -Method POST `
  -Uri http://127.0.0.1:8000/api/agents/recipe/recommend `
  -ContentType "application/json" `
  -Body '{"user_id":"demo","meal_type":"dinner"}'
$response.Headers["x-trace-id"]
```

用 trace_id 在日志中搜索完整链路。

### Swagger UI

访问 `http://127.0.0.1:8000/docs` 直接在浏览器测试 API。

---

## 第 2 步：Agent 执行了吗？

### Agent Chat 任务状态

```powershell
# 发送聊天
$result = Invoke-RestMethod -Method POST `
  -Uri http://127.0.0.1:8000/api/agent/chat `
  -ContentType "application/json" `
  -Body '{"message":"你好","user_id":"demo"}'

# 查询任务状态
Invoke-RestMethod "http://127.0.0.1:8000/api/agent/tasks/$($result.task_id)"
```

任务状态：`running` / `completed` / `failed`

失败时 `output_payload.error` 包含错误原因。

### 数据库查任务日志

```sql
SELECT id, task_type, status, input_payload, output_payload, created_at
FROM agent_tasks
ORDER BY created_at DESC
LIMIT 10;

SELECT tool_name, result_status, latency_ms, error_detail, trace_id
FROM tool_call_logs
ORDER BY created_at DESC
LIMIT 10;
```

## 第 3 步：Graph 节点有没有走？

### 单元测试 Graph

```powershell
.\.conda\python.exe -m pytest tests/test_agent_graphs.py -v
```

测试文件注入了 Mock Service，可验证 Graph 节点顺序和输出。

### Recipe Graph 节点顺序

```
collect_context → compose → check_equipment → finalize
```

如果厨具校验结果缺失，检查 `agent_graphs.py` 中 `_build_recipe_graph()` 的边定义。

### Inventory Graph 节点顺序

```
normalize_stage → intent_stage → response_stage
```

如果返回 fallback 回复，说明 `classify_intent()` 中 LLM 失败或意图为 unknown。

## 第 4 步：Service 有没有返回？

### 直接测试 Service

```python
# 在 Python REPL 中
from app.services.equipment_agent_service import EquipmentAgentService
from app.schemas.agent_tools import EquipmentCheckRequest

service = EquipmentAgentService()
result = service.check(EquipmentCheckRequest(
    equipment_owned=["锅"],
    required_equipment=["锅", "烤箱"],
))
print(result)
```

Service 层测试不依赖 HTTP 或 Graph，是最快的隔离调试方式。

### 运行示例脚本

```powershell
.\.conda\python.exe examples/call_equipment_agent.py
.\.conda\python.exe examples/call_recipe_agent.py
```

## 第 5 步：DB / ES / Redis 有没有数据？

### PostgreSQL

```powershell
docker compose exec postgres psql -U sebastian -d sebastian -c "SELECT count(*) FROM inventories;"
```

### Elasticsearch

```powershell
Invoke-RestMethod http://localhost:9200/memory_index/_count
```

### Redis

```powershell
docker compose exec redis redis-cli ping
```

## 第 6 步：LLM 调用成功了吗？

### 检查 API Key

```powershell
# 确认 .env 中 DEEPSEEK_API_KEY 已设置
Select-String "DEEPSEEK_API_KEY" .env
```

### 单独测试 LLM

```python
from app.llm.client import get_llm_client
client = get_llm_client()
print(client.chat([{"role": "user", "content": "说你好"}]))
```

### 常见 LLM 错误

| 日志/错误 | 原因 | 解决 |
|-----------|------|------|
| `401` | API Key 无效 | 检查 `.env` |
| `429` | 限流 | 等待或换 Key |
| `timeout` | 网络慢 | 调高 `LLM_TIMEOUT_MS` |
| `invalid JSON` | LLM 输出格式不对 | 检查 Prompt |
| `fallback to template` | LLM 失败已降级 | 正常行为，检查 WARNING 日志 |

## 日志

### 日志配置

- 级别：`LOG_LEVEL` 环境变量（默认 `INFO`）
- 配置：`app/core/logging_utils.py`
- 格式：结构化 JSON（trace_id、path、method、status_code、latency_ms）

### 关键日志点

| 位置 | 日志内容 |
|------|----------|
| `main.py` middleware | 每个请求的访问日志 |
| `nodes.py` | Intent 分类失败 WARNING |
| `recipe_agent_service.py` | LLM 降级 WARNING |
| Celery worker | 任务执行日志 |

### 提高日志级别

```env
LOG_LEVEL=DEBUG
```

## 前端联调

1. 确认 `frontend/.env.development` 中 `VITE_API_BASE_URL` 正确
2. 确认 `.env` 中 `CORS_ORIGINS` 包含前端地址
3. 浏览器 DevTools → Network 查看 API 请求和响应
4. 检查响应头 `x-trace-id`
