# Sebastian 智能菜谱 Agent 系统

> 5 分钟了解：这个项目是干什么的？

## 项目目标

Sebastian 是一个面向个人厨房与生活场景的 **多 Agent 智能饮食助手后端**。它不是传统的 CRUD 项目，而是一个 **AI Agent + 工作流编排 + 服务分层** 的系统。

主要能力：

| 能力 | 说明 | 入口 |
|------|------|------|
| 菜谱生成 | 根据餐食类型、热量目标、饮食偏好生成菜谱 | `RecipeAgent` |
| 食材分析 | 库存查询、调整、临期提醒 | `InventoryAgent` + CRUD API |
| 健康评估 | BMI 计算与饮食建议 | `HealthAgent` |
| 库存管理 | PostgreSQL 事务存储 | `InventoryService` |
| 厨具检查 | 对比所需厨具与用户拥有厨具 | `EquipmentAgent` |
| 网络/记忆搜索 | Elasticsearch 混合检索 | `SearchAgent` |

## 核心 Agent 关系

```
                    Recipe Agent
                         |
         +---------------+---------------+
         |               |               |
   Search Agent    Equipment Agent   Inventory Service
   (记忆检索)       (厨具校验)         (库存上下文)
         |
   Health Agent          Inventory Agent (聊天入口)
   (独立工具)            (LangGraph 意图编排)
```

- **Recipe Agent** 是最复杂的链路：编排层会先调用 Search 收集记忆上下文，再生成菜谱，最后调用 Equipment 校验厨具。
- **Inventory Agent** 走独立的 LangGraph 主图（`app/orchestration/graph.py`），用于 `/api/agent/chat` 自然语言对话。
- **Health / Equipment / Search** 各自是单步子图，通过 `/api/agents/*` 暴露。

## 技术栈一览

| 组件 | 技术 |
|------|------|
| API 框架 | FastAPI |
| Agent 编排 | LangGraph |
| LLM | DeepSeek（默认 `deepseek-chat`） |
| 事务数据库 | PostgreSQL |
| 记忆检索 | Elasticsearch |
| 缓存 / 队列 | Redis |
| 异步任务 | Celery |
| 前端 | React + TypeScript + Vite |

## 这是什么类型的项目？

新人需要建立的第一个认知：

> **这是一个 AI Agent 后端，不是普通 CRUD 项目。**

- 请求可能经过 **LangGraph 多节点编排**，而不是简单的 Controller → Service。
- 业务逻辑分布在 **Agent 入口、Orchestration 图、Service、Repository** 四层。
- LLM 调用有 **超时、重试、降级** 策略（例如 Recipe 在 LLM 失败时回退到模板生成）。
- 数据分 **PostgreSQL（事务真相源）** 和 **Elasticsearch（检索/记忆）** 两层。
