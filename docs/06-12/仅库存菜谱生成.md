# 仅库存菜谱生成

> 更新时间: 2026-06-12

## 1. 设计动机

原菜谱推荐 (`POST /api/agents/recipe/recommend`) 的生成流程包含三个子步骤：

1. **搜索记忆** — 从 Elasticsearch 检索用户饮食偏好和历史
2. **LLM 生成** — 基于偏好和餐食类型生成菜谱
3. **厨具检查** — 验证用户厨具是否满足菜谱需求

这个流程适合"开放式推荐"，但有一类场景不需要搜索记忆和厨具检查：**用户只想用当前库存里的食材做菜**。此时应该：

- 跳过搜索记忆（用户意图明确：用现有食材）
- 跳过厨具检查（简化流程，由 LLM 在 prompt 中处理）
- 以真实库存为唯一约束，强制 LLM 仅使用库存食材

## 2. 请求模型

### `InventoryOnlyRecipeRequest`

与 `RecipeRecommendRequest` 字段完全一致，但语义上表示"仅使用库存材料生成"：

```python
class InventoryOnlyRecipeRequest(BaseModel):
    user_id: str          # 必填, 1-64 字符
    meal_type: Literal["breakfast", "lunch", "dinner", "snack"]  # 默认 dinner
    target_calories: int  # 200-2000, 默认 600
    available_equipment: list[str]  # 用户拥有的厨具
    dietary_preferences: list[str]  # 饮食偏好
```

## 3. API 端点

### `POST /api/agents/recipe/recommend-from-inventory`

**请求示例:**

```json
{
  "user_id": "test-inv-only-001",
  "meal_type": "dinner",
  "target_calories": 600,
  "available_equipment": ["pan", "pot", "rice_cooker"],
  "dietary_preferences": ["high-protein"]
}
```

**成功响应 (200):** 返回标准 `RecipeRecommendResponse`，`ingredients` 中的食材均在用户库存中。

**错误响应:**

| 状态码 | 场景 | 响应 `error` 字段 |
|--------|------|-------------------|
| 400 | 用户库存为空 | `"库存为空，无法仅使用库存材料生成菜谱。请先添加库存食材。"` |
| 503 | LLM 不可用 | `"DeepSeek API Key 未配置"` 等 |

## 4. 实现架构

### 4.1 服务层

`app/services/recipe_agent_service.py` → `recommend_from_inventory_only()`

```python
def recommend_from_inventory_only(self, payload: InventoryOnlyRecipeRequest):
    # 1. 查询用户真实库存
    all_items = self._inventory_service.list_items(user_id=payload.user_id)
    if not all_items:
        raise ValidationError("库存为空...")

    # 2. 构造库存清单文本
    inventory_text = "\n".join(
        f"- {item.name}：{item.quantity} {item.unit}（过期日 {item.expire_date}）"
        for item in all_items
    )

    # 3. 调用 LLM（带库存约束 prompt）
    return self._recommend_with_inventory_constraint(payload, inventory_text)
```

### 4.2 LLM Prompt 核心约束

```
【核心约束】
你必须仅使用下方「当前库存」中列出的食材来生成菜谱。
不得推荐任何未在库存清单中出现的食材。
如果库存中缺少制作该菜谱所必需的关键食材，
仍需生成菜谱但必须在 missing_ingredients 字段中列出缺失的关键食材名称。
如果完全无法组合出任何合理的菜谱，请在 title 中写明「无法生成」，
在 rationale 中说明原因，ingredients 留空。
```

### 4.3 编排层

`app/orchestration/agent_graphs.py` → `run_recipe_agent_inventory_only()`

与标准 `run_recipe_agent()` 的区别：

| 维度 | 标准模式 | 仅库存模式 |
|------|---------|-----------|
| 搜索记忆子图 | ✅ 先搜索用户偏好 | ❌ 跳过 |
| 厨具检查子图 | ✅ 验证厨具 | ❌ 跳过 |
| 图节点数 | 4 个 (collect→compose→check→finalize) | 1 个 (execute) |
| 库存约束 | 不强制 | LLM prompt 强制 |

## 5. 边界处理

### 5.1 库存为空

`recommend_from_inventory_only()` 在第一步就校验库存是否为空，空则直接抛出 `ValidationError`（API 层返回 400）。

### 5.2 库存不足

LLM 允许在 `missing_ingredients` 中列出缺少的食材，同时尽力生成菜谱。极端情况下 `title` 为 `"无法生成"`。

### 5.3 食材用量

LLM prompt 要求"食材用量不得超过库存数量"，但这是 LLM 层面的软约束，不保证 100% 精确。确认制作时的库存扣减由 `InventoryService.batch_adjust_by_ingredients()` 做硬校验。

| 场景 | user_id | 说明 |
